# 下一阶段整体流程优化需求

当前系统已经支持 Docker 部署、数据库、日报生成和邮件推送。下一阶段不要增加新的选股策略，重点优化整体流程稳定性、数据一致性和可追踪性。

请完成以下设计和实现：

## 1. daily_report 增加 --mode both

当前 beginner 和 pro 会分别运行两次，导致重复抓数据、重复分析，且两份报告可能不一致。

请新增：

python -m analysis.daily_report --mode both --force

要求：
- 只抓取一次行情数据；
- 只运行一次市场分析、板块分析、情绪分析、观察池筛选；
- 同时生成 beginner 和 pro 两份报告；
- 保存路径保持：
  - reports/daily/daily_report_YYYYMMDD.md
  - reports/daily/daily_report_YYYYMMDD_pro.md
- 数据库 daily_report 表中分别写入两条记录；
- entrypoint.sh 改成调用 --mode both。

## 2. 生成 daily_summary_YYYYMMDD.json

在 daily_report 运行时，额外保存一份结构化摘要：

reports/daily/daily_summary_YYYYMMDD.json

内容包括：
- trade_date
- market
- sentiment
- themes
- quality
- watchlists 每个观察池 Top 若干只
- risk_directions
- generated_at

该 JSON 供 email_sender.py 使用，避免解析 Markdown。

## 3. email_sender.py 优先读取 summary.json

邮件推送逻辑改为：
- 优先读取最新 daily_summary_YYYYMMDD.json；
- 如果 summary.json 不存在，再降级解析 Markdown；
- 邮件正文使用 summary.json 生成摘要；
- beginner/pro markdown 报告继续作为附件。

## 4. 将观察池信号写入 stock_signal 表

daily_report 生成观察池后，把每个观察池里的股票写入 stock_signal。

要求：
- 同一天、同策略、同股票避免重复写入；
- 如果重复则更新；
- 保存字段包括：
  - trade_date
  - code
  - name
  - strategy
  - signal_type
  - hot_board_hits
  - close_price
  - pct_chg
  - volume_ratio
  - turnover
  - ma5/ma10/ma20
  - pct_5d/pct_20d
  - observe_low/observe_high
  - pressure_price
  - invalid_price
  - risk_level
  - action_signal
  - entry_reasons
  - risk_reasons

## 5. 新增 job_run_log 表和运行日志记录

新增表 job_run_log，用于记录每次任务运行：

字段：
- job_name
- trade_date
- status
- started_at
- finished_at
- duration_seconds
- error_message

至少记录：
- board_history
- daily_report
- email_sender

## 6. 新增 analysis/preflight_check.py

用于部署前检查：

检查：
- DATABASE_DSN 是否配置；
- 数据库能否连接；
- 必要表是否存在；
- stock_board_map 是否有数据；

运行：
python -m analysis.preflight_check

Docker 中：
docker compose run --rm --entrypoint "" stock-report python -m analysis.preflight_check

## 7. docker-compose.yml 增加 stock-mapper service

新增一个 service：

  image: stock-ai-system:latest

以后可以直接运行：
docker compose run --rm stock-mapper

## 8. 新增 Makefile

提供常用命令：
- make build
- make init-db
- make preflight
- make mapper
- make daily
- make email
- make logs

## 9. 不要做的事

- 不要新增复杂交易策略；
- 不要重构全部代码；
- 不要引入 Celery/Airflow/APScheduler；
- 不要删除现有 beginner/pro 报告；

目标：把系统从“能自动跑”升级为“稳定、可追踪、可复盘”的日报系统。

# 实现方案

## 改动文件和要点

### 1. `analysis/daily_report.py` — 核心

**`--mode` 增加 `both`**：
- `choices=["beginner", "pro", "both"]`
- `--mode both` 时抓一次数据、跑一次分析，分别调 `render_beginner_report` 和 `render_pro_report`
- 保存两份报告，DB 写两条 `daily_report` 记录

**新增 `build_summary_json(trade_date, market, sentiment, themes, quality, selectors)`**：
- 构造结构化 dict，保存为 `reports/daily/daily_summary_YYYYMMDD.json`
- 字段：trade_date, market, sentiment, themes, quality, watchlists(每池 Top N), risk_directions, generated_at

**新增 `save_stock_signals(selector_result, trade_date, db_conn)`**：
- 遍历每个观察池每只股票
- `INSERT INTO stock_signal (...) ON CONFLICT (trade_date, code, strategy) DO UPDATE`
- 写入字段：trade_date, code, name, strategy, signal_type, hot_board_hits, close_price, pct_chg, volume_ratio, turnover, ma5/10/20, pct_5d/20d, observe_low/high, pressure_price, invalid_price, risk_level, action_signal, entry_reasons, risk_reasons

**新增 `log_job_run(job_name, trade_date, db_conn)`**：
- 返回一个 context manager 或 start/end 函数对
- 写入 job_run_log 表（job_name, trade_date, status, started_at, finished_at, duration_seconds）
- 至少记录 board_history / daily_report / email_sender

### 2. `analysis/email_sender.py`

- `find_latest_summary()`：在 `reports/daily/` 找最新 `daily_summary_*.json`
- `build_email_body_from_json(summary)`：从 JSON 组装邮件正文
- `main()` 优先读 JSON，找不到则降级 markdown
- 附件不变

### 3. `sql/schema.sql`

新增 `job_run_log` 表：
```sql
CREATE TABLE IF NOT EXISTS job_run_log (
    id SERIAL PRIMARY KEY,
    job_name VARCHAR(100) NOT NULL,
    trade_date DATE,
    status VARCHAR(20) NOT NULL DEFAULT 'running',
    started_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    finished_at TIMESTAMP,
    duration_seconds NUMERIC,
    error_message TEXT
);
```

### 4. `analysis/init_db.py`

TABLES 列表加 `"job_run_log"`

### 5. `entrypoint.sh`

```bash
echo "=== 生成报告 ==="
python -m analysis.daily_report --mode both --force

echo "=== 推送消息 ==="
# PUSH_CHANNEL 判断不变（email / feishu / none）
```

### 6. `docker-compose.yml`

新增 stock-mapper service，复用同一镜像，entrypoint 覆盖为 `python -m analysis.stock_board_mapper`，volumes 和 env 与 stock-report 一致

### 7. `analysis/preflight_check.py` — 新文件

8 项检查，每项输出 `[OK]` / `[WARN]` / `[FAIL]`：
1. DATABASE_DSN 是否配置
2. 数据库能否连接
3. 5 张必要表是否存在
4. stock_board_map 是否有数据
5. SMTP 配置（SMTP_HOST/USER/PASSWORD/EMAIL_TO）是否完整
6. reports/ 和 logs/ 目录是否可写
7. 系统时区是否为 Asia/Shanghai
8. AkShare 基本连通性（取 1 只股票测试）

最后输出汇总：X OK / Y WARN / Z FAIL

### 8. `Makefile` — 新文件

```makefile
.PHONY: build init-db preflight mapper daily email logs

build:
	docker compose build

init-db:
	docker compose run --rm --entrypoint "" stock-report python -m analysis.init_db

preflight:
	docker compose run --rm --entrypoint "" stock-report python -m analysis.preflight_check

mapper:
	docker compose run --rm stock-mapper

daily:
	docker compose run --rm stock-report

email:
	docker compose run --rm --entrypoint "" stock-report python -m analysis.email_sender

logs:
	docker compose logs -f
```

## 不改的文件

- `feishu_sender.py` — 保留
- `selector.py` — 不增策略
- `report_renderer.py` — 不改
